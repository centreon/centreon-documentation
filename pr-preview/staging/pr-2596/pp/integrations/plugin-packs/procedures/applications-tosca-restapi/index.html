Tosca Rest API

Pack assetsâ€‹

Templatesâ€‹

The Monitoring Connector Tosca Rest API brings a host template:

  • App-Tosca-Restapi-custom

The connector brings the following service templates (sorted by the host template they are attached to):

  • App-Tosca-Restapi-custom
  • Not attached to a host template

By default, no services are created automatically using this template.

Service AliasService TemplateService Description
Execution-List-StatusApp-Tosca-Restapi-Execution-List-StatusCheck execution list status
Scenario-StatusApp-Tosca-Restapi-Scenario-StatusCheck scenario status

The services listed above are not created automatically when a host template is applied. To use them, create a service manually, then apply the service template you want.

Prerequisitesâ€‹

To monitor this type of resource, a user with read privileges on the Tosca Commander REST Webservice API is required.

Installing the monitoring connectorâ€‹

Packâ€‹

  1. If the platform uses an online license, you can skip the package installation instruction below as it is not required to have the connector displayed within the Configuration > Monitoring Connectors Manager menu. If the platform uses an offline license, install the package on the central server with the command corresponding to the operating system's package manager:
  • Alma / RHEL / Oracle Linux 8
  • Alma / RHEL / Oracle Linux 9
  • Debian 11
  • CentOS 7
dnf install centreon-pack-applications-tosca-restapi
dnf install centreon-pack-applications-tosca-restapi
apt install centreon-pack-applications-tosca-restapi
yum install centreon-pack-applications-tosca-restapi
  1. Whatever the license type (online or offline), install the Tosca Rest API connector through the Configuration > Monitoring Connectors Manager menu.

Pluginâ€‹

Since Centreon 22.04, you can benefit from the 'Automatic plugin installation' feature. When this feature is enabled, you can skip the installation part below.

You still have to manually install the plugin on the poller(s) when:

  • Automatic plugin installation is turned off
  • You want to run a discovery job from a poller that doesn't monitor any resource of this kind yet

More information in the Installing the plugin section.

Use the commands below according to your operating system's package manager:

  • Alma / RHEL / Oracle Linux 8
  • Alma / RHEL / Oracle Linux 9
  • Debian 11
  • CentOS 7
dnf install centreon-plugin-Applications-Tosca-Restapi
dnf install centreon-plugin-Applications-Tosca-Restapi
apt install centreon-plugin-applications-tosca-restapi
yum install centreon-plugin-Applications-Tosca-Restapi

Using the monitoring connectorâ€‹

Using a host template provided by the connectorâ€‹

  1. Log into Centreon and add a new host through Configuration > Hosts.
  2. Fill the Name, Alias & IP Address/DNS fields according to your ressource's settings.
  3. Apply the App-Tosca-Restapi-custom template to the host. A list of macros appears. Macros allow you to define how the connector will connect to the resource, and to customize the connector's behavior.
  4. Fill in the macros you want. Some macros are mandatory.
MacroDescriptionDefault valueMandatory
TOSCAAPIUSERNAMEAPI usernameX
TOSCAAPIPASSWORDAPI passwordX
TOSCAAPIPORTPort used80
TOSCAAPIPROTOCOLSpecify HTTPS if neededhttp
TOSCAEXTRAOPTIONSAny extra option you may want to add to every command (E.g. a --verbose flag). All options are listed here--insecure
  1. Deploy the configuration. The host appears in the list of hosts, and on the Resources Status page. The command that is sent by the connector is displayed in the details panel of the host: it shows the values of the macros.

Using a service template provided by the connectorâ€‹

  1. If you have used a host template and checked Create Services linked to the Template too, the services linked to the template have been created automatically, using the corresponding service templates. Otherwise, create manually the services you want and apply a service template to them.
  2. Fill in the macros you want (e.g. to change the thresholds for the alerts). Some macros are mandatory (see the table below).

  • Scenario-Status

EXTRAOPTIONSAny extra option you may want to add to the command (E.g. a --verbose flag). All options are listed here
  1. Deploy the configuration. The service appears in the list of services, and on Resources Status page. The command that is sent by the connector is displayed in the details panel of the service: it shows the values of the macros.

How to check in the CLI that the configuration is OK and what are the main options for?â€‹

Once the plugin is installed, log into your Centreon poller's CLI using the centreon-engine user account (su - centreon-engine). Test that the connector is able to monitor a server using a command like this one (replace the sample values by yours):

/usr/lib/centreon/plugins/centreon_tosca_restapi.pl \
    --plugin=apps::tosca::restapi::plugin \
    --mode=execution-list-status \
    --hostname='10.0.0.1' \
    --port='80' \
    --proto='http' \
    --api-username='myuser' \
    --api-password='*****'  \
    --workspace='MyWorkspace' \
    --execution-list-id='24c429d8-a6d7-46a7-a057-ac08c391d76f'

The expected command output is shown below:

OK: Execution list 'Centreon' entries passed: 1, failed: 0, not executed: 0, unknown: 0 | 'entries.passed.count'=1;;;0; 'entries.failed.count'=0;;0:0;0; 'entries.not_executed.count'=0;;;0; 'entries.unknown.count'=0;;;0;

Troubleshootingâ€‹

Please find the troubleshooting documentation for the API-based plugins in this chapter.

Available modesâ€‹

All available modes can be displayed by adding the --list-mode parameter to the command:

/usr/lib/centreon/plugins/centreon_tosca_restapi.pl \
    --plugin=apps::tosca::restapi::plugin \
    --list-mode

The plugin brings the following modes:

ModeLinked service template
execution-list-statusApp-Tosca-Restapi-Execution-List-Status
scenario-statusApp-Tosca-Restapi-Scenario-Status

--extend-perfdata-groupAdd new aggregated metrics (min, max, average or sum) for groups of metrics defined by a regex match on the metrics' names. Syntax: --extend-perfdata-group=regex,namesofnewmetrics,calculation[,[ne wuom],[min],[max]] regex: regular expression namesofnewmetrics: how the new metrics' names are composed (can use $1, $2... for groups defined by () in regex). calculation: how the values of the new metrics should be calculated newuom (optional): unit of measure for the new metrics min (optional): lowest value the metrics can reach max (optional): highest value the metrics can reach Common examples: Sum wrong packets from all interfaces (with interface need --units-errors=absolute): --extend-perfdata-group=',packets_wrong,sum(packets_(discard |error)_(in|out))' Sum traffic by interface: --extend-perfdata-group='traffic_in_(.*),traffic_$1,sum(traf fic_(in|out)_$1)'Output
